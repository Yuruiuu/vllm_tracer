# 实验环境
硬件平台：AutoDL 云服务器，NVIDIA GeForce RTX 4090 D

软件环境：Python 3.10.8、PyTorch 2.11.0+cu130、CUDA 13.0、vLLM 0.23.0

测试模型：Qwen/Qwen2.5-7B-Instruct-AWQ（AWQ 量化版本）

# 一、基于 PyTorch Profiler 的性能分析
通过设置环境变量`VLLM_TORCH_PROFILER_DIR` 开启 vLLM 内置的 PyTorch Profiler 支持，执行`vllm bench latency`基准测试命令，生成 Perfetto 格式的性能追踪数据，通过 TensorBoard 与 Perfetto UI 进行多维度分析。

鉴于平台显存限制，使用AWQ 量化的 Qwen2.5-7B 模型降低显存占用。

## 核心发现

### GPU 执行侧特征
GPU 侧总耗时 1.011s，算子耗时分布如下：

AWQ GEMM 算子：占比 62.99%，为最核心的 GPU 耗时来源，符合量化推理的计算特征；

FP16 GEMM 算子：占比约 19%；

FlashAttention 算子：占比约 4%。

整体来看，GPU 算力主要消耗在矩阵乘法运算，无异常冗余算子，计算负载符合大语言模型推理的典型特征。但 GPU 整体利用率与 SM 效率偏低，算力未充分释放。

### CPU 调度侧为核心瓶颈
CPU 侧总耗时 2.107s，约为 GPU 耗时的 2 倍，调度开销占比极高：

cudaLaunchKernel（内核启动）耗时占 CPU 总时间的 21.14%；

框架调度逻辑（如vllm::unified_attention_with_output）占比 14%；

cudaMemcpyAsync（异步内存拷贝）占比 10%。

## PyTorch Profiler 的局限性
在分析过程中发现，PyTorch Profiler 仅面向算子层面，无法适配 vLLM 推理服务的全链路分析需求.
- 缺乏请求维度关联：仅能统计全局 Kernel 耗时，无法将 GPU 算子与上层请求、调度步骤对应，无法回答 “该 Kernel 属于哪个请求的哪个推理阶段”；
- 时间体系不统一：GPU Kernel 为相对时间，CPU 侧事件为系统时间，两套时间基准无法直接对齐，难以分析调度与执行的因果关系；
- 服务层指标缺失：无法直接输出首 Token 延迟（TTFT）、单 Token 耗时（TPOT）、吞吐量等推理服务核心指标；
- 多进程适配性差：vLLM v1 采用多进程架构，PyTorch Profiler 无法跨进程关联调度与执行数据。

# 二、vllm_tracer
设计vllm_tracer性能剖析工具，核心目标：
- 实现请求维度的全链路追踪，覆盖服务层、运行时层、算子层三层架构；
- 统一全局时间基准，实现三层事件的时间轴精准对齐；
- 支持 Perfetto 原生可视化，无需额外开发前端界面；
- 适配 vLLM v1 多进程架构，保障探针有效生效。
工具采用三层分层架构，所有事件基于统一时间基准、携带请求 ID，最终通过统一事件总线聚合导出。
```
┌─────────────────────────────────────────────────────┐
│              统一时间轴（perf_counter_ns）           │
├─────────────────────────────────────────────────────┤
│ Layer 3 服务层：TTFT/TPOT/吞吐量/（扩展KV Cache指标 ）│  聚合指标
│ Layer 2 运行时层：队列→Prefill→Decode→Sampling       │  Hook调度
│ Layer 1 算子层：CUDA Kernel/内存拷贝                 │  算子采集
└─────────────────────────────────────────────────────┘
        ↓ 统一事件总线（EventBus） ↓
    导出Chrome Trace格式 → Perfetto UI可视化
```
（1）Layer 1 算子层（Kernel Layer）
对接 PyTorch Profiler 采集 GPU 侧算子、内存拷贝等事件，核心工作是将 GPU 相对时间校准为全局绝对时间，实现与上层事件的时间轴对齐。

（2）Layer 2 运行时层（Runtime Layer）
通过 Monkey Patch Hook vLLM 核心调度与执行模块（Scheduler、GPUModelRunner），捕获 Prefill、Decode 等调度事件，每个事件携带request_id，实现请求粒度的调度追踪。

（3）Layer 3 服务层（Serving Layer）
基于运行时层事件聚合计算服务级核心指标，包括首 Token 延迟、单 Token 平均耗时、系统吞吐量、KV 缓存使用率等，提供端到端的性能评估。

## 初步验证与分析价值

在 AWQ 量化版 Qwen2.5-7B 模型上运行 vLLM 官方 latency benchmark，对生成的 Trace 数据进行 Perfetto 可视化分析，验证了三层统一时间轴方案的可行性。

vllm_tracer 已完成服务层、运行时层与算子层的三层性能追踪框架设计与原型实现，并通过统一时间基准实现跨层事件对齐。工具基于 Runtime Hook 与 PyTorch Profiler 采集调度事件和 GPU Kernel 事件，统一汇聚至 EventBus，最终导出 Perfetto Trace 进行可视化分析。

初步实验表明：

- 实现了 Request → Runtime → GPU Kernel 的统一时间轴关联追踪；
- 能够观察 Continuous Batching 的调度过程及请求间执行重叠关系；
- 支持 Decode Step 级别的性能分析；
- 实现了 GPU Kernel 与 Runtime 调度事件的统一时间轴对齐；
- 能够从请求视角分析 TTFT、TPOT、Batch 执行过程及系统吞吐特征。

相比传统 PyTorch Profiler 仅提供算子级视图，vllm_tracer 进一步建立了请求、调度与 GPU 执行之间的关联关系，为大模型推理系统性能分析提供了跨层观测能力。

## 当前局限

当前版本已完成三层事件贯通，但仍处于原型验证阶段。

主要局限包括：

- 算子层数据仍依赖 PyTorch Profiler，采集开销较大；
- GPU 侧仅支持 Kernel 时间信息，尚未获取 SM Occupancy、Tensor Core 利用率、显存带宽等硬件指标；
- Request 与 Kernel 的关联主要基于时间对齐，尚未建立精确的因果映射关系；
- 尚未覆盖 PagedAttention、KV Cache、Sampling 等关键推理组件的细粒度事件；
- 当前验证环境为单机共享 GPU 场景，多进程与分布式场景仍需进一步验证。

后续将探索基于 CUPTI 的低开销采集方案，并扩展至“请求→调度→算子→硬件”的完整性能分析链路。

Request → Runtime → CUPTI Activity → GPU Hardware
